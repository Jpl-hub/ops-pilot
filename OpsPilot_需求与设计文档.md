# OpsPilot-X 需求与设计文档

## 1. 文档定位

本文档用于统一 OpsPilot-X 的产品目标、赛题对齐关系、系统边界、核心架构与实现方案，作为后续开发、交付、评审汇报与项目归档的基线文档。

项目名称：OpsPilot-X  
项目主题：智能体赋能的新能源企业运营分析与决策支持系统  
聚焦领域：新能源上市公司运营分析、风险识别与决策支持  
文档版本：V2.0  
更新日期：2026-03-24

本文档以当前仓库的真实实现为基础编写，不再沿用历史参考稿或概念稿表述。

## 2. 赛题解读与项目定位

根据《2026 年中国大学生计算机设计大赛大数据主题赛“智能体赋能的企业运营分析与决策支持系统”赛题说明》，作品需要满足以下核心要求：

1. 以智能体为核心载体，而不是只做静态看板或普通问答系统。
2. 围绕特定行业构建企业运营分析与决策支持能力，本项目选择新能源领域。
3. 面向不同用户提供差异化服务，至少覆盖投资者、企业管理者、监管/风控等角色。
4. 基于多源数据完成智能问数、运营评估、风险机会洞察、定制化内容生成等任务。
5. 强调数据整合、自然交互、自主分析、来源追溯、来源核验与逻辑拆解。
6. 允许补充合法合规数据，但必须以公开、合规、可说明来源的数据为基础。

结合赛题要求，本项目的定位不是“新能源财报聊天机器人”，而是一个三位一体的企业运营分析平台：

1. 传统应用层：提供清晰、稳定、可交互的业务系统界面与操作流程。
2. Agent 与大模型层：提供多角色、多工具、多轮协同分析能力。
3. 大数据工程层：提供真实数据采集、文档解析、结构化加工、质量巡检与可追溯证据链。

项目的核心目标是把“能问”“能算”“能核”“能解释”“能落地”整合为一个闭环系统。

## 3. 设计原则

### 3.1 真实数据优先

系统默认以正式数据目录作为唯一分析基础，不允许在正式链路缺失时静默回退为样本结果。  
只有显式开启 `OPS_PILOT_ALLOW_SAMPLE_FALLBACK=true` 时，才允许开发联调用的 bootstrap 数据参与结果构建。

### 3.2 可信分析优先

大模型负责理解问题、规划工具调用与组织答案，但不负责凭空编造指标、排名或证据。  
所有关键结论必须能落到以下至少一种可核查对象：

1. 官方财报页级证据
2. 结构化 silver 指标
3. 研报摘录或行业研究报告
4. 公式回放链路
5. 文档解析产物

### 3.3 产品交付优先

用户界面面向真实使用场景，不展示比赛口号、答辩话术、演示版语义。  
系统页面只保留业务功能、运行状态、数据质量和分析结果。

### 3.4 失败显式优先

关键能力未配置时必须显式阻断，不允许以伪结果、空泛总结、样本兜底替代真实分析。  
例如文本补证依赖模型鉴权时，如果缺少 API Key，系统直接报错而不是伪造回答。

## 4. 用户角色与核心需求

本项目围绕三类核心用户设计：

### 4.1 投资者

核心需求：

1. 快速理解公司经营质量与同业位置
2. 获取风险提示与机会信号
3. 核验研报观点与真实财报是否一致
4. 在复杂信息中快速定位原文证据

### 4.2 企业管理者

核心需求：

1. 获取经营体检结果与问题优先级
2. 找到现金、库存、应收、盈利质量等薄弱环节
3. 结合同业对标识别差距来源
4. 生成有执行顺序的治理动作建议

### 4.3 监管 / 风控角色

核心需求：

1. 批量筛查高风险企业
2. 识别异常信号与风险传播链
3. 对重点企业执行多轮跟踪与核验
4. 关注系统运行稳定性与数据质量

## 5. 项目目标与边界

### 5.1 总体目标

围绕新能源上市公司，构建一个面向多角色的企业运营分析与决策支持系统，覆盖以下能力：

1. 智能问数与多轮协同分析
2. 企业运营评估与问题诊断
3. 行业风险扫描与机会洞察
4. 研报观点核验与来源追溯
5. 图谱检索、压力测试、多模态文档解析
6. 数据质量治理与运行保障

### 5.2 当前边界

当前系统聚焦新能源上市公司，不泛化到所有行业。  
系统重点处理以下数据：

1. 上市公司定期报告
2. 个股与行业研报
3. 行业与宏观信号
4. 文档解析中间产物与结构化指标

### 5.3 非目标

以下内容不是当前版本的主目标：

1. 全行业统一评分模型
2. 完整量化交易系统
3. 离线本地大模型推理能力本身
4. 无数据支撑的开放式闲聊

## 6. 总体方案

系统采用“前端交互层 + 后端服务层 + 智能体协同层 + 数据工程层”的总体架构。

### 6.1 前端交互层

技术栈：Vue 3 + TypeScript + Vite

主要页面：

1. 首页：公开展示系统覆盖情况与核心入口
2. 协同分析：多轮问答、Agent 处理链、结果面板
3. 企业体检：五维评分、关键指标、风险标签、建议动作
4. 产业大脑：行业观察、运行信号、研究摘要
5. 风险预警：风险扫描与风险变化跟踪
6. 图谱检索：关系链路与节点关联分析
7. 压力测试：冲击场景下的传导模拟
8. 研报核验：观点与真实财报逐条对照
9. 多模态解析：文档升级、OCR 结果与结构回放
10. 证据页：原文片段、页码、来源文档、路径与指纹
11. 运营保障中心：数据覆盖、运行状态、OCR 质检与治理轨迹

### 6.2 后端服务层

技术栈：FastAPI + PostgreSQL + pgvector

后端职责：

1. 提供统一 API 接口
2. 管理认证、角色切换与会话
3. 承接业务分析服务
4. 执行智能体调度与工具调用
5. 管理文档解析与质量巡检
6. 提供运行周报与管理接口

### 6.3 智能体协同层

系统采用半开放式 Orchestrator 方案。  
智能体不是多个互相空转的“大脑”，而是一个可控的任务协同链：

1. 任务识别：识别问题意图、公司主体和报期
2. 数据分析：选择评分、核验、图谱、压力、检索等真实工具
3. 证据校验：回放证据、核查页码、组织引用
4. 策略生成：结合用户角色输出下一步动作建议

### 6.4 数据工程层

系统围绕 `raw -> bronze -> silver -> snapshot/vector` 建设数据流水线：

1. `raw`：抓取财报、研报、行业研究等原始数据
2. `bronze`：生成页级文本、块级切片、解析中间产物
3. `silver`：抽取结构化财务指标与企业快照
4. `snapshot`：形成批量风险扫描和运行态快照
5. `vector`：支持 Hybrid RAG 与检索增强

## 8. 功能设计

### 8.1 首页

首页承担两个职责：

1. 对外展示系统覆盖范围与核心能力
2. 作为登录前的公共入口

首页调用公开接口 `/api/v1/public/landing`，不依赖登录态，不再错误访问管理接口。

### 8.2 协同分析

协同分析是系统主入口。  
用户输入自然语言问题后，系统执行以下流程：

1. 识别角色、公司、报期与任务类型
2. 触发对应工具链
3. 对结果进行证据补强
4. 以结构化方式返回答案、证据、图表、建议动作与追问

支持的问题类型包括：

1. 企业评分
2. 同业对标
3. 风险扫描
4. 研报核验
5. 指标直查
6. 图谱查询
7. 压力测试
8. 简报生成

### 8.3 企业体检

企业体检围绕五维评分体系展开，当前覆盖 19 项指标。  
评分结果包含：

1. 总分与等级
2. 分维度得分
3. 风险标签
4. 弱项与机会项
5. 关键指标
6. 公式回放
7. 建议动作
8. 证据分组

### 8.4 同业对标

系统在同子行业公司池内进行横向比较，输出：

1. 公司排名
2. 总分对比图
3. 子行业对标列表
4. 同周期横向位置

### 8.5 风险预警与产业大脑

系统提供两个互补视角：

1. 风险预警：聚焦公司层面的风险标签与变化
2. 产业大脑：聚焦行业景气、政策、研究与运行信号

它们共同承担赛题中的“风险与机会洞察”要求。

### 8.6 研报核验

系统支持将研报观点与真实财报进行逐条核验，输出：

1. 匹配观点
2. 偏差观点
3. 缺乏支撑的观点
4. 预测卡片
5. 证据链接

### 8.7 图谱检索

图谱检索面向复杂关系问题，支持：

1. 企业关系下钻
2. 焦点节点追踪
3. 推理链展示
4. 文档与运行流关联

### 8.8 压力测试

系统面向极端情景下的经营冲击问题，提供：

1. 冲击等级判断
2. 传导阶段分析
3. 恢复动作序列
4. 场景推演日志

### 8.9 多模态解析与文档升级

系统围绕 OCR 标准链路建设文档升级能力，支持：

1. 版面还原
2. 标题恢复
3. 跨页拼接
4. 单元格追踪
5. 文档升级结果展示

### 8.10 运营保障中心

运营保障中心不是后台壳，而是系统工程能力的一部分。  
它负责暴露以下内容：

1. 覆盖诊断与漏斗
2. 系统就绪度
3. 运行时检查
4. OCR Contract 质量巡检
5. 关键检查清单
6. 治理轨迹与作业队列
7. 运行周报导出

## 9. Agent 与大模型设计

### 9.1 设计目标

Agent 层的设计目标不是展示“多智能体数量”，而是提高真实任务完成率。  
因此当前采用可控的半开放式调度方案，而不是完全自由对话式自治。

### 9.2 执行链路

一次典型分析请求的执行链如下：

1. 前端提交自然语言问题
2. Orchestrator 识别问题类型与公司
3. 按白名单选择真实工具
4. 获取结构化结果
5. 如果证据不足，则执行 Hybrid RAG 补证
6. 构造 answer sections、charts、action cards、evidence groups
7. 返回给前端并记录运行历史

### 9.3 真实工具优先

当前大模型可以调度但不限于以下工具能力：

1. 企业评分引擎
2. 同业对标引擎
3. 风险扫描器
4. 研报核验器
5. 图谱推理器
6. 压力模拟器
7. 指标路由器
8. Brief 生成器

### 9.4 可信机制

为防止“大模型说得对但没有依据”，系统加入以下约束：

1. 关键结果优先来自后端计算而非 LLM 自算
2. 证据条数不足时执行补证流程
3. 证据页必须落到 chunk、页码、来源标题和指纹
4. 公式卡必须展示指标回放关系
5. 无鉴权、无数据、无结果时直接显式失败

## 10. 大数据工程设计

### 10.1 数据来源

项目当前对齐赛题给出的公开下载路径，优先采用以下来源：

1. 财务报告：上交所、深交所、北交所
2. 研报数据：东方财富行业研报、个股研报
3. 宏观数据：国家统计局

如需补充其他数据，必须满足以下前提：

1. 来源公开、合法、可说明
2. 不违反数据安全与披露规定
3. 能在文档中清楚说明口径与用途

### 10.2 数据分层

#### Raw 层

保存原始 PDF、研报详情与原始清单。

#### Bronze 层

保存页级 JSON、块级切片、OCR 解析结果、文档升级中间产物。

#### Silver 层

保存结构化指标、公司快照、分析直接可用的数据对象。

#### Snapshot / Vector 层

用于风险扫描、快照回放与检索增强。

### 10.3 质量治理

为保证数据工程不是“拉下来就用”，系统引入质量治理：

1. 主周期覆盖率统计
2. 页级解析覆盖统计
3. silver 指标覆盖统计
4. 研报覆盖统计
5. OCR Contract 质检
6. 文档升级任务队列
7. 运行轨迹记录

### 10.4 正式数据优先策略

当前仓库已实现以下策略：

1. 正式数据仓优先
2. 默认禁止静默样本回退
3. 样本只作为显式联调开关使用

这项策略直接服务于赛题中的“可信分析”和项目交付中的“真实结果”要求。

## 11. 关键接口与系统形态

系统当前以 Web 应用 + API 服务形式交付，符合赛题允许的提交形态。

关键接口包括：

1. `/api/v1/chat/turn`：协同分析入口
2. `/api/v1/company/score`：企业评分
3. `/api/v1/company/benchmark`：同业对标
4. `/api/v1/industry/risk-scan`：行业风险扫描
5. `/api/v1/claim/verify`：研报观点核验
6. `/api/v1/company/stress-test`：压力测试
7. `/api/v1/evidence/{chunk_id}`：证据查看
8. `/api/v1/admin/overview`：运行保障概览
9. `/api/v1/admin/delivery-report`：运行周报导出
10. `/api/v1/public/landing`：公开首页指标

## 13. 安全、合规与风险控制

### 13.1 数据合规

系统优先接入赛题给定的公开数据源，不接入来源不明数据。

### 13.2 结果合规

系统输出定位于辅助决策，不声称替代审计、监管裁定或投资建议。

### 13.3 运行风险

当前主要运行风险包括：

1. OCR 标准链路未接通时，多模态能力受限
2. 正式数据覆盖不足时，部分公司无法进入完整分析
3. 数据更新频率不足时，主周期分析结果会滞后

因此系统通过运行时检查和运营保障中心持续暴露这些问题，而不是隐藏。

## 14. 当前版本的优势与后续强化方向

### 14.1 当前版本优势

1. 行业聚焦明确，新能源场景完整
2. 前端产品形态完整，不是单接口 Demo
3. Agent 协同链清晰，可解释性较强
4. 数据工程分层明确，具备 raw/bronze/silver 演进路径
5. 证据追溯、公式回放、研报核验形成可信闭环
6. 运营保障中心补足了系统工程与交付能力

### 14.2 后续强化方向

如果继续向更高评审档位推进，下一阶段应重点强化：

1. 扩大正式公司池与主周期覆盖率
2. 增强宏观数据与产业链数据接入
3. 提高 OCR 标准链路稳定性与表格解析精度
4. 优化多角色下的定制化报告生成质量
5. 增加更强的跨期趋势分析与行业传播分析

## 15. 结论

OpsPilot-X 的核心不是把“大模型”“Agent”“财报分析”简单拼在一起，而是围绕新能源企业运营分析场景，建立了一条从真实数据采集、文档解析、结构化加工、智能体协同、证据核验到业务交付的完整链路。

从赛题视角看，本项目已经形成以下统一能力：

1. 有行业聚焦
2. 有多角色服务
3. 有智能体主导分析
4. 有真实数据工程支撑
5. 有风险与机会洞察
6. 有来源追溯与核验机制
7. 有可交付的产品形态

因此，OpsPilot-X 的目标不是成为一个“会说话的财报页面”，而是成为一个真正可用、可核、可管、可交付的新能源企业运营分析与决策支持系统。
